# SatelliteOverpass 项目 C++ 编码规范

## 1. 文档目的

本文档定义了 SatelliteOverpass 项目的 C++ 编码规范，旨在确保代码风格统一、可读性强、易于维护和扩展。本规范基于 Google C++ Style Guide 和 CppCoreGuidelines，并结合项目特点进行了适当调整。

## 2. 适用范围

本规范适用于 SatelliteOverpass 项目的所有 C++ 代码，包括：
- 核心功能模块
- 测试代码
- 工具脚本
- 示例代码

## 3. 命名规范

### 3.1 通用原则

- 使用描述性名称，避免使用缩写（除非是广泛接受的缩写，如 `TLE`、`ECEF` 等）
- 名称应清晰表达变量、函数或类的用途
- 避免使用单个字母作为变量名（除了循环计数器和数学变量）

### 3.2 具体命名规则

| 项目 | 命名风格 | 示例 |
|------|---------|------|
| 类和结构体 | PascalCase | `CoordinateConverter`、`SatellitePassPredictor` |
| 函数和方法 | camelCase | `toGeodetic()`、`computeGST()` |
| 变量和参数 | camelCase | `semiMajor`、`geodeticPosition` |
| 常量 | kPrefix + CamelCase | `kEarthSemiMajor`、`kPi` |
| 枚举值 | kPrefix + CamelCase | `kSuccess`、`kFileNotFound` |
| 命名空间 | PascalCase | `SatelliteOverpass`、`Math` |
| 文件名 | PascalCase | `CoordinateSystem.h`、`SatelliteOverpass.cpp` |

### 3.3 特殊情况

- **宏**：全部大写，使用下划线分隔（如 `MAX_SATELLITES`）
- **模板参数**：单个大写字母或描述性名称（如 `T`、`ElementType`）
- **私有成员变量**：可选使用 `m_` 前缀（如 `m_semiMajor`）

## 4. 代码风格

### 4.1 缩进和空格

- 使用 4 个空格进行缩进（不使用制表符）
- 大括号 `{` 放在行尾，与控制语句或函数声明在同一行
- 小括号 `()` 内不添加空格
- 中括号 `[]` 内不添加空格
- 运算符两侧添加空格（如 `a = b + c`）
- 逗号后添加空格（如 `function(a, b, c)`）

### 4.2 行长度和换行

- 每行代码长度不超过 80 个字符
- 长行应在适当位置换行，换行后缩进 8 个空格
- 函数参数列表过长时，每个参数单独占一行

### 4.3 空行和注释

- 在函数定义之间添加空行
- 在逻辑块之间添加空行
- 注释应清晰、简洁，解释代码的意图而非实现细节
- 使用 `//` 进行单行注释，`/* */` 进行多行注释

## 5. 语言特性使用

### 5.1 现代 C++ 特性

- **优先使用 C++17/20 特性**：
  - `constexpr` 用于编译时计算
  - `std::array` 替代 C 风格数组
  - `std::vector` 用于动态数组
  - `std::string` 替代 C 风格字符串
  - `std::unique_ptr`/`std::shared_ptr` 替代裸指针
  - `std::format` 替代字符串拼接
  - `std::span` 替代指针 + 长度的接口
  - `[[nodiscard]]` 标记重要返回值
  - `noexcept` 标记不抛出异常的函数

### 5.2 异常处理

- 使用异常而非错误代码进行错误处理
- 异常应继承自 `SatelliteException` 或标准异常
- 异常消息应清晰描述错误原因
- 避免在性能关键路径上使用异常

### 5.3 模板和泛型

- 合理使用模板，避免过度泛化
- 使用 C++20 concepts 约束模板参数
- 模板代码应简洁明了，避免复杂的模板元编程

## 6. 性能相关规范

### 6.1 内存管理

- 优先使用 RAII 原则管理资源
- 避免频繁的内存分配和释放
- 对小对象考虑使用对象池或内存池
- 合理使用移动语义减少拷贝开销

### 6.2 计算优化

- 对频繁计算的常量使用 `constexpr` 或预计算
- 避免在循环中进行重复计算
- 合理使用内联函数（`inline`）
- 考虑使用 OpenMP 进行并行计算

### 6.3 算法选择

- 选择时间复杂度合适的算法
- 对大规模数据使用高效的数据结构
- 优先使用标准库算法而非手动实现

## 7. 测试代码规范

### 7.1 单元测试

- 测试文件命名：`test_<module_name>.cpp`
- 测试用例命名：`TEST(TestCaseName, TestName)`
- 测试用例应独立，不依赖其他测试的状态
- 测试覆盖率目标：核心功能 ≥ 80%

### 7.2 测试风格

- 使用 Google Test 框架
- 测试代码应简洁明了，专注于验证功能
- 测试用例应覆盖正常情况和边界情况
- 使用断言而非打印语句进行验证

## 8. 版本控制规范

### 8.1 提交信息

- 提交信息应清晰描述变更内容
- 使用 imperative 语气（如 "Add feature" 而非 "Added feature"）
- 提交信息格式：
  ```
  <类型>: <简短描述>

  <详细描述，可选>
  ```
- 类型包括：`feat`（新功能）、`fix`（修复）、`refactor`（重构）、`docs`（文档）、`test`（测试）、`chore`（构建/依赖）

### 8.2 分支管理

- `main` 分支：稳定版本
- `develop` 分支：开发版本
- 特性分支：`feature/<feature_name>`
- 修复分支：`fix/<issue_number>`

## 9. 代码审查规范

### 9.1 审查要点

- 代码是否符合本规范
- 功能是否正确实现
- 是否存在潜在的性能问题
- 是否存在内存泄漏或安全隐患
- 代码是否易于理解和维护

### 9.2 审查流程

1. 提交者创建 Pull Request
2. 审查者进行代码审查
3. 提交者根据反馈进行修改
4. 审查者确认修改后合并

## 10. 工具支持

### 10.1 静态分析工具

- **Clang-Tidy**：代码风格和潜在问题检查
- **Cppcheck**：静态代码分析
- **AddressSanitizer**：内存错误检测
- **UndefinedBehaviorSanitizer**：未定义行为检测

### 10.2 代码格式化工具

- **clang-format**：自动代码格式化，配置文件基于本规范

## 11. 附则

- 本规范应随着项目的发展和 C++ 标准的更新而不断完善
- 特殊情况下可以偏离本规范，但应在代码中添加注释说明原因
- 本规范的最终解释权归 SatelliteOverpass 项目维护者所有
